# TASS — Trusted AI Security Scanner
### Full Capability Reference · 2026-05-04

---

## What TASS Is

TASS is a GitHub App that watches every pull request for **newly introduced capabilities** in AI-generated or human-written code. When code is pushed, TASS:

1. Detects new external dependencies, HTTP calls, database writes, filesystem access, and privilege patterns using a two-layer engine (dependency diffing + Tree-sitter AST queries).
2. Diffs detected capabilities against a committed `tass.manifest.yaml` baseline to isolate **novel** ones.
3. Posts a PR comment and GitHub Check requiring a human decision on each novel capability.
4. Serves a hosted web UI where developers can **Confirm** or **Revert** each item.
5. Once all capabilities are decided, commits the updated manifest to the PR branch and marks the GitHub Check as passed.

The net effect: AI-generated code cannot silently acquire new external reach. Every new network call, new database, new file path, or new privilege pattern must be explicitly signed off by a human with write access to the repository.

## CLI Commands

### `tass init`

Scans the entire local repository and generates `tass.manifest.yaml`.

```
tass init [--path .] [--rules-dir ./rules]
```

- **Layer 0:** Parses `go.mod`, `requirements.txt`, `package.json`. Each package → one capability entry.
- **Layer 1:** Walks all source files, runs Tree-sitter queries. Each match → one capability entry.
- Writes a YAML file with one entry per detected capability, all marked `status: auto_detected`.
- Prints a summary breakdown: N deps, M AST matches, total capabilities.

**Use case:** First-time setup. Commit the generated manifest to establish the baseline. Future scans diff against it.

---

### `tass scan`

Diffs HEAD against a base branch and reports novel capabilities (present in HEAD but not in the committed manifest).

```
tass scan [--base main] [--format text|json] [--rules-dir ./rules]
          [--path .] [--ci] [--export-to URL] [--token TOKEN]
          [--repo owner/repo] [--branch feature-branch]
```

**Flags:**
| Flag | Default | Purpose |
|---|---|---|
| `--base` | `main` | Base branch to diff against |
| `--format` | `text` | Output format: human-readable or JSON |
| `--rules-dir` | (embedded) | Override rule directory |
| `--path` | `.` | Repository root |
| `--ci` | false | Emit GitHub Actions `::error::` annotations |
| `--export-to` | — | POST results to TASS dashboard URL |
| `--token` | — | Bearer token for `--export-to` |
| `--repo` | auto-detect | `owner/repo` for export |
| `--branch` | auto-detect | Branch name for export |

**Exit codes:** 0 = no novel capabilities. 1 = novel capabilities found (blocks CI).

**Air-gap export:** `--export-to` POSTs the capability JSON to the hosted server, which creates a scan entry and returns a verify URL. Useful when the scanner runs in CI but the TASS server isn't the GitHub App (private repos, offline CI).

---

```

```

|---|---|

| `TASS_GITHUB_CLIENT_ID` | GitHub OAuth App client ID |

| Variable | Purpose |
|---|---|

| `TASS_IMPORT_TOKEN` | Shared secret for `POST /api/import` |

On startup: runs SQLite migrations, initialises all subsystems, begins listening. On SIGINT/SIGTERM: 10-second graceful shutdown.

### `tass seed`

Inserts realistic demo data for presentations.

```
tass seed [--db tass.db]
```

Inserts:
- 1 installation (`acme-corp`)
- 2 repositories (`payments-service`, `ai-support-bot`)
- 5 scan results spread over the past 2 weeks
- 18 verification decisions across 3 developers (`alice`, `bob`, `carol`)
- Carol intentionally has >50% revert rate (useful for demo of override-rate analytics)

Idempotent — safe to run multiple times, skips if demo data already exists.

---

### `tass policy`

Generates a security policy document from the committed `tass.manifest.yaml`.

```
tass policy [--format k8s|iam] [--app myapp] [--namespace default]
            [--path .] [--output policy.yaml]
```

- **`k8s`:** Kubernetes `NetworkPolicy` allowing egress to DNS (53) and HTTPS (443) for pods matching the app label. Annotated with `tass.dev/generated-from` and `tass.dev/generated-at`.
- **`iam`:** AWS IAM Policy JSON with `Allow` statements derived from detected capabilities (e.g., `s3:*` for S3 access, wildcarded ARNs where specific resources are unknown).

**Use case:** Translate the TASS manifest into enforceable infrastructure policy without manual mapping.

---

### `tass version`

Prints version, commit SHA, and build date (injected at build time via Makefile `ldflags`).

## Detection Engine

### Layer 0 — Dependency Diffing

Supported manifest files:

| File | Parser |
|---|---|
| `go.mod` | `GoModParser` |
| `requirements.txt` | `RequirementsTxtParser` |
| `package.json` | `PackageJSONParser` |

Each parser extracts `{package_name: version}` maps. For a diff scan, TASS fetches the base branch version of the file and computes additions. Each new package becomes one `external_dependency` capability with confidence 0.90.

**Capability ID format (Layer 0):** `dep:{language}:{package_name}`

---

### Layer 1 — Tree-sitter AST Queries

TASS uses Tree-sitter grammars for Go, Python, and JavaScript to run structural pattern matching against source code. Rules are `.scm` (S-expression query) files paired with `.meta.yaml` metadata files.

**Rule loading precedence:**
1. Explicit `--rules-dir` flag
2. Local `./rules/` directory (if present on disk)
3. Embedded rules bundled into the binary at build time

**Capability ID format (Layer 1):** `ast:{language}:{cap_id}:{captured_symbol}`

Example: a Tree-sitter match on `http.Post(...)` in Go → `ast:go:net/http:client:Post`

**Directories skipped:** `.git`, `vendor`, `node_modules`, `__pycache__`, `.venv`, `venv`, `dist`, `build`, `.cache`

**Path exclusions:** `.tassignore` in the repo root (gitignore syntax) can exclude paths from scanning.

---

### Capability Categories

| Category | Badge Color | Meaning |
|---|---|---|
| `external_dependency` | Purple | New package added to a manifest file |
| `network_access` | Blue | Outbound HTTP, TCP, gRPC, OTEL export |
| `database_operation` | Orange | SQL query, Redis, ORM, NoSQL write |

## Database Schema

All state lives in a single SQLite file (WAL mode). Migrations are applied automatically on `tass serve` startup.

### `installations`
One row per GitHub App installation (org or user account).

```sql
id INTEGER PRIMARY KEY          -- GitHub installation ID
account_login TEXT              -- GitHub org/user login
account_type TEXT               -- "Organization" | "User"
installed_at DATETIME
access_token TEXT               -- Cached installation token
token_expires_at DATETIME
```

### `repositories`
One row per repository within an installation.

```sql
id INTEGER PRIMARY KEY
installation_id INTEGER → installations(id)
full_name TEXT                  -- "owner/repo"
default_branch TEXT             -- "main"
manifest_sha TEXT               -- SHA of last committed tass.manifest.yaml
created_at DATETIME
```

### `scan_results`
One row per TASS scan triggered by a PR event.

```sql
id TEXT PRIMARY KEY             -- UUID
repo_id INTEGER → repositories(id)
installation_id INTEGER
pr_number INTEGER
head_branch TEXT
commit_sha TEXT
base_sha TEXT
scanned_at DATETIME
scan_duration_ms INTEGER
capabilities_json TEXT          -- JSON array of Capability objects
novel_count INTEGER
status TEXT                     -- "pending" | "verified" | "expired"
check_run_id INTEGER            -- GitHub Check Run ID
comment_id INTEGER              -- GitHub PR comment ID
```

### `verification_decisions`
One row per human decision (Confirm or Revert) on a capability in a scan.

```sql
id TEXT PRIMARY KEY
scan_id TEXT → scan_results(id)
capability_id TEXT              -- Matches Capability.ID
decision TEXT                   -- "confirm" | "revert"
justification TEXT              -- Optional free-text from developer
decided_by TEXT                 -- GitHub login
decided_at DATETIME
UNIQUE(scan_id, capability_id)  -- One decision per cap per scan (INSERT OR REPLACE)
```

### `manifest_history`
Immutable record of every manifest snapshot committed to a PR branch.

```sql
id TEXT PRIMARY KEY
repo_id INTEGER → repositories(id)
commit_sha TEXT                 -- SHA of the commit that wrote the manifest
content_yaml TEXT               -- Full YAML content
committed_by TEXT               -- GitHub login of who triggered the commit
committed_at DATETIME
```

### `audit_events`
Tamper-evident hash-chain audit log. Every significant action writes a row here.

```sql
id TEXT PRIMARY KEY
ts DATETIME DEFAULT now()
tenant_id INTEGER               -- installation_id (multi-tenant scope)
actor_gh_id INTEGER             -- GitHub user numeric ID
actor_login TEXT                -- GitHub username
repo TEXT                       -- "owner/repo"
action TEXT                     -- Action constant (see Audit Actions below)
target_id TEXT                  -- Subject of the action (scan ID, cap ID, etc.)
before_json TEXT                -- State before (capability metadata only, no source code)
after_json TEXT                 -- State after
ip TEXT                         -- Client IP
user_agent TEXT
prev_hash TEXT                  -- SHA-256 of previous event in chain
hash TEXT                       -- SHA-256 of this event (includes prev_hash)
```

**Indexed on:** `(tenant_id, ts)`, `(tenant_id, repo, ts)`

## Authentication & RBAC

### OAuth Flow
1. Unauthenticated request to any protected page → 302 redirect to `/auth/github?return_to=...`
2. `/auth/github` redirects to GitHub authorization endpoint (`scope: read:user repo`)
3. GitHub redirects to `/auth/github/callback?code=...&state=...`
4. Callback exchanges code for access token, creates HMAC-signed session cookie, redirects to `return_to`

### Session
Sessions are HMAC-SHA256-signed cookies (no server-side storage). The session stores:
- `GitHubLogin` — GitHub username
- `AvatarURL` — Profile picture URL
- `AccessToken` — GitHub OAuth access token (used for RBAC checks)
- `CreatedAt` — For optional expiry

### Permission Model
| Action | Required permission |
|---|---|
| View verify page | Any authenticated user |
| Submit Confirm/Revert | `write` or higher collaborator access |
| Invoke `/approve` or `/reject` slash commands | `write` or higher |
| View dashboard | Any authenticated user |
| View audit trail | Any authenticated user |
| Export CSV | Any authenticated user |

Permissions are fetched from GitHub's Collaborators API and cached for 5 minutes per `login@owner/repo` key.

## GitHub App Integration

### JWT Authentication
TASS generates short-lived JWTs signed with the GitHub App private key (RSA PKCS1 or PKCS8 PEM). These are exchanged for installation-specific access tokens (cached until expiry).

### Events handled
| Event | Action | What TASS does |
|---|---|---|
| `pull_request` | `opened`, `synchronize`, `reopened` | Triggers full scan pipeline |
| `installation` | `created` | Scans all added repos, opens setup PRs |
| `installation_repositories` | `added` | Scans newly added repos |
| `issue_comment` | `created` | Checks for `/approve` or `/reject` slash commands |

### Scan Pipeline (on PR event)
1. Fetch `tass.manifest.yaml` from the base branch via Contents API
2. Fetch dependency files changed in the PR
3. Fetch source files changed in the PR
4. Run Layer 0 dep diff → novel dependency capabilities
5. Run Layer 1 AST scan on changed source files → novel AST capabilities
6. Diff all detected capabilities against the committed manifest
7. If novel capabilities exist:
   - Create GitHub Check Run (`action_required`)
   - Create or update PR comment with capability table and verify link
   - Store scan result in SQLite
8. If zero novel capabilities:
   - Create GitHub Check Run (`success`, "No novel capabilities detected")

### Slash Commands
PR comments containing `/approve` or `/reject` (from users with write+ access) are parsed by `SlashCommandHandler` and bulk-applied to all pending capabilities on the associated PR's scan.

## Manifest Format

`tass.manifest.yaml` — committed to the repository root.


## Tamper-Evident Audit Log

Every decision, scan trigger, manifest commit, policy export, permission denial, and slash command invocation is written to `audit_events` as an append-only hash chain.

### Hash chain mechanism
- Each event includes `prev_hash`: the SHA-256 of the previous event in the same `tenant_id` chain.
- `hash` = SHA-256(id + ts + tenant_id + action + prev_hash + all other fields)
- Any modification to a historical event breaks the chain from that point forward.
- `GET /api/audit/chain/verify?tenant_id=N` recomputes and validates the entire chain.

### Compliance mapping
| Standard | Controls |
|---|---|
| SOC 2 | CC7.2 (system monitoring), CC7.3 (evaluation of anomalies) |
| ISO 27001 | A.8.15 (logging), A.8.16 (monitoring activities) |
| NIST 800-53 | AU-2 (event logging), AU-3 (content of records), AU-9(3) (cryptographic protection), AU-10(3) (non-repudiation), AU-12 (audit generation) |

**Data minimisation:** TASS never writes customer source code to `before_json` or `after_json`. Only capability metadata (IDs, names, categories, decisions) is logged.

## Complete Request Flow

**1. Developer opens a PR**

GitHub fires a `pull_request.opened` webhook → `POST /webhooks/github`.

Handler validates HMAC-SHA256 signature. Returns 202. Background goroutine calls `pipeline.ScanFunc()`.

**2. Scan pipeline executes**

```
Pipeline.ScanFunc()
  ├── Fetch tass.manifest.yaml from base branch (GitHub Contents API)
  ├── scanner.ScanDiff(repoRoot, baseBranch)
  │     ├── Layer 0: diff go.mod/requirements.txt/package.json → new packages
  │     └── Layer 1: AST scan changed .go/.py/.js files → matched rules
  ├── manifest.Diff(detectedSet, committedManifest) → novel only
  └── if novel > 0:
        App.CreateCheckRun(status="action_required")
        App.CreatePRComment(capability table + /verify/{scan-id})
        store.SaveScan(scanResult)
      else:
        App.CreateCheckRun(status="success", "No novel capabilities")
```

**3. Developer reviews capabilities**

Opens the verify link in the PR comment → `GET /verify/{scan-id}`.

`VerifyPageHandler` fetches the scan result and existing decisions, renders `verify.templ`.

Developer sees capability cards. Clicks Confirm or Revert.

**4. HTMX decision submission**

```
POST /ui/verify
  body: scan_id=uuid&capability_id=ast:go:net/http:client:Post&decision=confirm&justification=...

UIVerifyHandler
  ├── Extract GitHub login from session
  ├── verifier.Decide(scanID, capID, decision, justification, login)
  │     ├── store.SaveDecision(VerificationDecision)
  │     ├── audit.SaveAuditEvent(capability.confirmed, hash chain)
  │     ├── store.GetDecisionsByScan(scanID) → check if all decided
  │     └── if allDecided:
  │           manifest.Diff + manifest.Apply + manifest.Marshal → YAML
  │           App.CommitFile(branch, "tass.manifest.yaml", yaml)
  │           store.SaveManifestSnapshot(commitSHA, yaml)
  │           store.UpdateScanStatus("verified")
  │           App.UpdateCheckRun(status="success")
  │           audit.SaveAuditEvent(manifest.committed)
  └── Returns HTMX HTML fragments: updated card + progress bar (OOB) + maybe success banner (OOB)
```

**5. GitHub Check passes → PR can merge**

`tass.manifest.yaml` on the PR branch now includes the newly decided capabilities. When the PR merges, the manifest becomes the new baseline for future scans.

## Key Types Reference

```go
// pkg/contracts
type Capability struct {
    ID          string        // Deterministic: "dep:go:stripe" or "ast:go:net/http:client:Post"
    Name        string
    Category    CapCategory   // "network_access", "database_operation", etc.
    Source      DetectionLayer // "layer0_dependency" | "layer1_ast"
    Location    CodeLocation  // File, Line, Column
    Confidence  float64       // 0.0–1.0
    RawEvidence string        // Matched code snippet or package name
}

// pkg/manifest
type ManifestEntry struct {
    ID            string
    Name          string
    Category      CapCategory
    Source        DetectionLayer
    Status        string        // "auto_detected" | "confirmed" | "reverted"
    ConfirmedBy   string
    ConfirmedAt   *time.Time
    FirstDetected *time.Time
    Note          string
    Locations     []CodeLocation
}

// internal/storage
type ScanResult struct {
    ID              string
    RepoID          int64
    InstallationID  int64
    PRNumber        int
    HeadBranch      string
    CommitSHA       string
    BaseSHA         string
    ScannedAt       time.Time
    ScanDurationMS  int64
    Capabilities    []contracts.Capability  // Full set detected (not just novel)
    NovelCount      int
    Status          ScanStatus
    CheckRunID      int64
    CommentID       int64
    FullName        string  // Populated via JOIN with repositories
}

type VerificationDecision struct {
    ID            string
    ScanID        string
    CapabilityID  string
    Decision      string  // "confirm" | "revert"
    Justification string
    DecidedBy     string
    DecidedAt     time.Time
}

// internal/audit
type AuditEvent struct {
    ID          string
    Ts          time.Time
    TenantID    int64
    ActorGHID   int64
    ActorLogin  string
    Repo        string
    Action      Action
    TargetID    string
    BeforeJSON  string
    AfterJSON   string
    IP          string
    UserAgent   string
    PrevHash    string  // SHA-256 of previous event
    Hash        string  // SHA-256 of this event
}
```
